## 5. Authentication & Authorization

### 5.1 Auth flow

1. `POST /auth/register` → BCrypt-hashes password, creates user
2. `POST /auth/login` → returns JWT
3. JWT used in `Authorization: Bearer <token>` on all subsequent calls
4. Gateway validates JWT before routing (centralized — Phase E)
5. `GET /auth/validate` exposed for gateway/service token verification

### 5.2 JWT payload structure

- **Subject** = username
- **Custom claim:** `role`
- **IssuedAt + Expiration** timestamps
- **Algorithm:** HS256
- **No userId in token** — services use username for identity

### 5.3 JwtUtil.java methods (auth-service)

```
generateToken(username, role)  → returns signed JWT
extractUsername(token)         → from subject claim
extractRole(token)             → from "role" custom claim
isValid(token)                 → validates signature + parses claims
```

### 5.4 Frontend auth state

- localStorage key: `ax_token`
- On 401 response: token cleared, `returnTo` saved in sessionStorage as `returnTo` (path + query), redirect to `/login`
- Axios request interceptor adds `Authorization: Bearer <token>` to every request
- `axiosConfig.js` baseURL: `import.meta.env.VITE_API_BASE_URL` (= `http://localhost:9090/api/v1`)

## 9. Internal Service Logic

### 9.1 Switch flow (transaction-service `SwitchService.java`)

**Authorize:**
1. Get terminal info via Feign (TerminalServiceClient)
2. Enrich merchant context (MCC, region) via Feign (MerchantServiceClient) — graceful degradation if merchant service down
3. Verify open batch exists for terminal — else throws "No open batch"
4. Call risk-service via Feign (`/risk/check?amount=&panMasked=&tid=`) — defaults to ALLOW if risk service down
5. Build AuthMessage with: terminal info, merchant info, masked PAN (via `MaskingUtil.maskPan`), network = `LOCALSIM` (hard-coded)
6. Authorization decision:
   - If risk = `BLOCK` → status DECLINED, responseCode `05`
   - Else if amount ≤ ₹50,000 → status APPROVED, authCode `A<timestamp>`, responseCode `00` (REVIEW logs warning but still approves)
   - Else → status DECLINED, responseCode `51` (limit exceeded)
7. Save AuthMessage

**Void:**
- Original must be APPROVED + SALE
- Open batch required
- Original status set to REVERSED
- New AuthMessage created with txnType=VOID, authCode `V<timestamp>`, responseCode `00`

**Refund:**
- Original must be APPROVED
- Refund amount cannot exceed original
- Open batch required
- New AuthMessage created with txnType=REFUND, authCode `R<timestamp>`, responseCode `00`

**Networks:** All transactions use `LOCALSIM` network (no real Visa/MC/UnionPay routing)

### 9.2 Fee calculation (transaction-service `FeeService.java` + `FeeRuleMatcher.java`)

**Components:** `schemeFee`, `interchangeFee`, `acquirerMarkup` → sum = `totalFee` → `netMerchantAmount = amount - totalFee`

**Rule matching (FeeRuleMatcher):**
- Rule must be `ACTIVE`
- Within effective date window (`effectiveFrom` ≤ now ≤ `effectiveTo`)
- MCC pattern: trailing wildcard supported (e.g., `54*` matches `5411`, `5499`); else exact match
- Region: case-insensitive exact match
- Amount: between minAmount and maxAmount inclusive
- Network: case-insensitive exact match
- Null criteria = no constraint
- All AND-combined

**Calculation:**
- BigDecimal with HALF_UP rounding, 4 decimal places
- All matching rules' percentages summed per component
- Rules sorted by `priority` ASC
- If no rules match → all fees zero (`FeeBreakdownDTO.empty`)

**`createTxnFromAuth(authId)` flow:**
- Auth must be APPROVED — else `IllegalStateException`
- Enrich merchant MCC/region if missing on auth
- Calculate fees → create Txn with `settled=false`

**Settlement support methods:**
- `getUnsettledTxns(merchantId)` — used by settlement-service
- `markTxnsSettled(merchantId)` — used by settlement-service

### 9.3 Risk decisioning (risk-service `RiskService.java`)

**`checkRisk(amount, panMasked, tid)` flow:**
1. **Terminal blacklist check** — if TID in active blacklist → `BLOCK, score=100, "Terminal is blacklisted"`
2. **PAN blacklist check** — normalize PAN via `MaskingUtil.maskPan`, if in blacklist → `BLOCK, score=100, "Card is blacklisted"`
3. **Active rules loop** — for each active rule, if `amount > rule.maxAmount`:
   - Calculate score: `min((amount / maxAmount) * 50, 100)`
   - If rule's action is more severe than current worst → escalate
4. Severity ranking: BLOCK (3) > REVIEW (2) > ALLOW (1)
5. Reason format: `"Rule triggered: <name> (amount <X> exceeds limit <Y>)"`

**Important:** The `expression` field on `risk_rule` is **stored but NOT actually evaluated** — it's metadata only. Real logic uses `maxAmount` field comparison.

**`evaluateAndSave(txnId, amount)` — called after transaction:**
- For each rule that triggered (amount > maxAmount), saves a RiskEvent

**Blacklist normalization:** PAN values are normalized via `MaskingUtil.maskPan()` before storing or matching

### 9.4 Settlement scheduler (`SettlementScheduler.java`)

- **Cron:** `0 0 1 * * *` — runs daily at 1:00 AM
- **Flow:**
  1. Fetch all active merchants via Feign (page size 1000)
  2. Loop each merchant, call `settlementService.settle(merchantId)`
  3. Track counters: success / skipped (`IllegalStateException` = no unsettled txns) / failed (other exceptions)
- `runManualSettlement()` exists for ad-hoc triggering

### 9.5 Idempotency

- **Header:** `Idempotency-Key`
- **Supported endpoints:** `/transactions/authorize`, `/transactions/void`, `/transactions/refund`, `/txns/from-auth/{authId}`
- **Storage:** `acquirerx_transaction.idempotency_record` (key, response, created_at, expires_at)
- **Cleanup:** `IdempotencyCleanupScheduler.java` — scheduled job removes expired records

### 9.6 PAN masking

- All PANs stored masked (e.g., `699006X*X*4178`)
- `PanMaskingConverter` (JPA AttributeConverter) in transaction-service
- `MaskingUtil.maskPan()` utility — used in transaction-service AND risk-service
- Logging: `PanMaskingConverter` logger filter masks PAN in log output

### 9.7 Audit logging (auth-service `AuditService.java`)

- **Method:** `logAction(actorUsername, action, targetType, targetId, details)`
- **Storage:** `acquirerx_auth.audit_log`
- **Sortable fields:** auditId, actorUsername, action, targetId, performedAt
- **Filter:** by actorUsername, action, targetId, fromDate, toDate

## 12. Frontend Error Handling

### 12.1 `errorMapper.js` — `mapError(err)` priority order

1. **No response (network error)** → `ECONNABORTED` = "request timed out", else "cannot reach server"
2. **errorCode lookup** — if `data.errorCode` or `data.code` matches MESSAGES dictionary → return mapped message
3. **Backend message** — return as-is UNLESS it's generic (filtered list: "internal server error", "bad request", "unauthorized", "forbidden", "not found")
4. **HTTP status fallback:**
   - 401 → "You are not authorised to perform this action"
   - 403 → "You don't have permission for this resource"
   - 404 → "The requested record could not be found"
   - 409 → "A record with these details already exists"
   - 422 → "Some fields are invalid"
   - 5xx → "Something went wrong on our side"
5. **Generic fallback** — "Something went wrong. Please try again."

### 12.2 Mapped error codes (28 total)

**Auth:** INVALID_CREDENTIALS, ACCOUNT_LOCKED, TOKEN_EXPIRED, UNAUTHORIZED, FORBIDDEN
**Validation:** VALIDATION_FAILED, DUPLICATE_RESOURCE, RESOURCE_NOT_FOUND
**Merchant:** MERCHANT_KYC_PENDING, MERCHANT_INACTIVE, MID_ALREADY_EXISTS
**Terminal:** TID_ALREADY_EXISTS, TERMINAL_INACTIVE, NO_OPEN_BATCH
**Transaction:** TXN_ALREADY_VOIDED, TXN_ALREADY_REFUNDED, TXN_ALREADY_SETTLED, REFUND_AMOUNT_EXCEEDS, RISK_BLOCKED
**Settlement:** BATCH_ALREADY_CLOSED, NOTHING_TO_SETTLE
**Dispute:** DISPUTE_DEADLINE_PASSED, DISPUTE_ALREADY_CLOSED
**Recon:** RECON_FILE_INVALID, RECON_ITEM_ALREADY_RESOLVED
**Infrastructure:** SERVICE_UNAVAILABLE, GATEWAY_TIMEOUT, INTERNAL_ERROR, NETWORK_ERROR

### 12.3 `shouldToastError(err)` rules

- Skip 401 (handled by interceptor redirect)
- Skip `ERR_CANCELED` (request abort)
- Toast everything else

### 12.4 Per-call suppression

Add `suppressToast: true` to axios request config to silently fail (used for stats endpoints with known backend issues).
